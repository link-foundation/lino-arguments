# Case Study: npm Trusted Publishing E404 Error — Workflow Filename Mismatch

## Issue Reference

- **Issue**: [#31 - CI/CD publishing for JavaScript is broken](https://github.com/link-foundation/lino-arguments/issues/31)
- **CI Run**: [24265468331](https://github.com/link-foundation/lino-arguments/actions/runs/24265468331/job/70859460577)
- **Date**: 2026-04-10

## Executive Summary

The npm package `lino-arguments` failed to publish via GitHub Actions with OIDC trusted publishing due to a **workflow filename mismatch**. The npm trusted publisher configuration on npmjs.com was set to trust workflow `main.yml`, but the repository had been refactored to use `js.yml` as the JavaScript CI/CD workflow. npm's OIDC verification rejects tokens from unregistered workflow files with a misleading E404 error.

Additionally, the `publish-to-npm.mjs` script had a bug where it reported success (`✅ Published`) even when `changeset publish` failed, because `@changesets/cli` exits with code 0 even when individual package publishes fail.

## Root Cause Analysis

### Primary Root Cause: Workflow Filename Mismatch

npm's OIDC trusted publishing verifies that the OIDC token was generated by a specific workflow file. The verification chain is:

1. GitHub Actions generates an OIDC token with claims including:
   - `repository`: `link-foundation/lino-arguments`
   - `workflow_ref`: `link-foundation/lino-arguments/.github/workflows/js.yml@refs/heads/main`
2. npm checks the token's `workflow_ref` against its trusted publisher configuration
3. npm's config expects: `.github/workflows/main.yml`
4. Token contains: `.github/workflows/js.yml`
5. **Mismatch → npm returns 404** (not 403, because npm uses 404 to avoid leaking package existence info)

From the existing case study screenshot (`docs/case-studies/npm-config-screenshot.png`):

| npm Setting        | Value                          |
| ------------------ | ------------------------------ |
| Trusted Publisher  | `link-foundation/lino-arguments` |
| **Workflow**       | **`main.yml`**                 |

Current workflow file: `.github/workflows/js.yml`

### Secondary Bug: False Success Report

The `publish-to-npm.mjs` script had a critical bug:

```javascript
// Bug: changeset publish exits with code 0 even when packages fail
await $`npm run changeset:publish`;
// This line executes even on publish failure:
setOutput('published', 'true');
console.log(`✅ Published ${PACKAGE_NAME}@${currentVersion} to npm`);
```

`@changesets/cli` logs errors but exits with code 0, so the script's try/catch never catches the failure. The script now verifies by querying the npm registry after publish.

### Tertiary Issue: Stale Version Publish Attempt

The workflow has a `check_publish` step that detects `package.json` version `0.3.0` is not on npm and triggers a publish attempt. This is a valid recovery mechanism, but when combined with the workflow filename mismatch, it leads to repeated failed attempts.

## How npm OIDC Trusted Publishing Works

```
GitHub Actions                    npm Registry                npmjs.com Settings
┌─────────────┐                   ┌──────────────┐            ┌──────────────────┐
│ js.yml      │──OIDC token──────▶│ Verify OIDC  │◀───────────│ Trusted Publisher│
│ workflow    │  claims:          │ token claims │  Expected:  │ Workflow: main.yml│
│             │  workflow=js.yml  │              │  main.yml   │                  │
└─────────────┘                   └──────┬───────┘            └──────────────────┘
                                         │
                                    MISMATCH!
                                    ▼ 404 Error
```

### Key Constraint

> **npm only allows ONE workflow file as trusted publisher per package.**

This is documented in the reference repositories:
- [hive-mind](https://github.com/link-assistant/hive-mind) — uses `release.yml`
- [js-ai-driven-development-pipeline-template](https://github.com/link-foundation/js-ai-driven-development-pipeline-template) — uses `release.yml`

## Solution

### Required Fix 1: Update npm Trusted Publisher Configuration (Manual)

The npm package owner must update the trusted publisher workflow on npmjs.com:

1. Go to https://www.npmjs.com/package/lino-arguments/access
2. Under "Trusted Publisher", click "Edit"
3. Change "Workflow" from `main.yml` to `js.yml`
4. Save

### Required Fix 2: Post-Publish Verification (Code — Applied)

The `publish-to-npm.mjs` script now verifies publication by querying the npm registry after `changeset publish` completes, instead of trusting the exit code.

### Required Fix 3: Workflow Documentation (Code — Applied)

Added a comment to `js.yml` documenting the npm trusted publisher workflow filename requirement.

## Lessons Learned

1. **npm OIDC trusted publishing is tied to a specific workflow filename** — renaming the workflow file breaks publishing without any clear error message (404 is misleading).

2. **Always verify publish success** — `@changesets/cli` exits with code 0 even when packages fail to publish. Scripts must verify by querying the npm registry.

3. **Document workflow constraints** — When a workflow file has external dependencies (npm OIDC config), document this clearly in the file itself.

4. **404 ≠ "package not found"** — npm uses 404 for permission/authorization failures to avoid leaking information. This is a known pattern in npm's security model.
